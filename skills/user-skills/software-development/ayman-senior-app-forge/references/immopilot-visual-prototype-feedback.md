# ImmoPilot Visual Prototype Feedback — compact modern ERP UI

Session lesson from clickable MVP prototype work.

## What happened

- V1 prototype was functionally useful for judging ERP scope, but Ayman disliked the oversized scrollable sidebar.
- Ayman expected a more modern/creative SaaS dashboard direction, closer to an airy rounded reference dashboard screenshot: compact navigation, polished cards, whitespace, soft borders/shadows, and a friendlier top-level composition.
- V2 was routed to Kimi with explicit instruction to keep ImmoPilot serious while modernizing the shell.

## Durable UI rule

For ImmoPilot visual prototypes and major ERP screens:

- Do not default to a large scrollable enterprise sidebar.
- Prefer compact horizontal top navigation, section tabs, or a thin non-scroll rail.
- All primary modules should be reachable without sidebar scrolling on desktop.
- Keep the interface serious and finance-aware, but give it modern SaaS polish: airy canvas, rounded cards, strong typography, calm chips/pills, polished spacing, and subtle icons.
- Use creative composition as a product-direction aid, not as decorative Dribbble filler.

## What to copy from modern references

- Airy light canvas with white surfaces.
- Compact nav pills and segmented controls.
- Rounded cards with soft borders/shadows.
- Big but useful greeting/command header.
- Asymmetric but balanced dashboard grids.
- Subtle avatars/icons/status chips.
- Clear hierarchy and generous whitespace.

## What not to copy

- The other product's domain model, labels, or metrics.
- Toy charts or decorative visualizations that do not map to real estate ERP decisions.
- Crypto/AI gradients, emoji-heavy icons, or generic dashboard sludge.
- Pure beauty that cannot scale into an operating ERP.

## Prototype workflow implication

When Ayman asks for a product-direction prototype, produce at least two visual directions when feasible:

1. serious ERP baseline, optimized for scope clarity;
2. modern/compact SaaS direction, optimized for taste and product feel.

Both should stay clickable and should preserve the same domain modules so Ayman can judge style without losing functional coverage.
